# Privacy Boundary — 資料 AI 邊界判斷

主管 AI 共學課最常被問的問題。**不要講技術**（API / SOC2 / 加密）。用 3 條白話規矩。

## 何時啟動

用戶說：
- 「我能不能把 X 給 Claude」
- 「客戶資料能不能餵 AI」
- 「資安疑慮怎麼降到最低」
- 「訂閱戶 email 可以丟嗎」
- 「個資 / PII / GDPR / 機密 / 內部資料」
- 「老闆會不會擔心」
- 「公司資料能用 AI 嗎」
- 「會不會洩漏」
- 「Claude 會記住嗎」

## 3 條白話規矩（核心）

### 規矩 1: 公開資料 — 全 OK ✅
- 已經在你公司網站 / 公開報告 / 媒體訪問裡的內容
- 例：產品介紹文案、公開 KPI、新聞稿、行銷貼文
- **直接餵，不用想**

### 規矩 2: 內部資料 — 脫敏後 OK ⚠️
- 公司內部但**不識別到特定客戶 / 員工**的東西
- 例：銷售總額（不含客戶名）、流程 SOP、團隊架構
- **動作**：把人名 / 客戶名 / 郵箱 / 電話 / 身分證 取代成 `客戶A` / `員工X` / `email_001` 再餵
- 替代：把資料先複製到 Excel，用「尋找取代」把識別欄位清掉

### 規矩 3: 客戶 PII / 個資 — 不要直餵 ❌
- 客戶 email / 電話 / 身分證 / 地址 / 訂單明細含真實姓名
- 健康 / 財務 / 家庭 / 信用相關
- **替代方案 A**：餵「假資料」結構讓 AI 寫程式 / 寫 SQL，你自己跑（資料不離開你電腦）
- **替代方案 B**：餵 1 筆完全脫敏的「樣本」讓 AI 給 template，你套到真資料上
- **不確定 = 不要餵**，問講師 / 公司資安窗口

## 主管常見場景對照

| 場景 | 屬於 | 動作 |
|------|------|------|
| 寫行銷文案 | 規矩 1 | 直接 |
| 整理本月銷售報告（含真客戶名）| 規矩 3 | 客戶名換代號再餵 |
| 寫 follow-up 信草稿（含客戶姓名）| 規矩 3 | 寫「敬愛的 客戶X」AI 給 template |
| 訂閱戶資料分析 | 規矩 3 | 餵假資料結構讓 AI 寫 SQL |
| 編採訪逐字稿（公開人物）| 規矩 1 | 直接 |
| 內部 SOP 整理 | 規矩 2 | 部門特定但無客戶識別，直接 |
| 員工績效面談記錄 | 規矩 3 | 不餵，AI 只能幫 framework |
| HR 招募初篩 CV | 規矩 3 | 候選人姓名脫敏 |

## 為什麼這樣設計

主管 unique value = **設 guardrail 給屬下**。不是自己會用 AI，是會教屬下「什麼不要餵」。

3 條規矩主管能背下來、能教屬下、能寫進 W9 部門 AI 使用規範。

## 不要做

- ❌ 講 SOC2 / Anthropic API retention policy / encryption（主管不在乎）
- ❌ 一律「不要餵任何敏感資料」（過度保守，主管用不下去）
- ❌ 不給替代方案就拒絕（主管會去問 ChatGPT）
- ❌ 用「視情況而定」打發（給 3 條明確規矩）
