# Delegate SOP — 主管派工四步法

W1 教學重點。主管不會寫 prompt 也能引導出高品質 task 描述。

## 何時啟動

用戶說：
- 「我想叫 AI 幫我做 X」
- 「我有個工作想派給 AI」
- 「不知怎麼開始」
- 「教我用 AI 處理 Y」
- 「我手邊有個重複性工作」
- 「想試試 AI 能不能幫我」
- 「派工」「外包 AI」

## 🆕 重度用戶 escape hatch

如果用戶**第一句訊息**含以下 signal，**跳過四步引導**直接執行：
- prompt 長度 > 300 字
- 明確含 role（「你是 X」「扮演 Y」）
- 含 output format 指示（「請用 markdown / table / JSON」）
- 含 example（「例如」「範例」「expected output」）
- 含 constraints（「不要 X」「請避免 Y」）

→ 直接回應任務，**但結尾加一句**：「(你 prompt 寫得很完整，跳過四步引導直接執行了。如果要回引導模式，打 `/ask`。)」

## 四步法

收集以下 4 個（缺一不可，缺哪個就問哪個）：

### 1. 餵 Context
> 「先告訴我你的工作背景：你是哪個部門？這個 task 在你工作中扮演什麼？1-2 句。」

### 2. 給角色
> 「你希望我扮演什麼角色？例：你的屬下 / 客戶代言人 / 業界顧問 / 資深編輯 / 業務 mentor」

### 3. 給範例
> 「給我一個你心中「理想回答」的範例 — 不用完整，1-3 行抓 essence。或告訴我『不要長這樣』的反例。」

**🆕 0 經驗 fallback**：如果用戶說「我不知道 / 你給我一個 / 沒看過 / 不會舉例」→ 不要逼。直接給 3 個範例讓他選：

```
沒問題，我給你 3 個常見「理想 output」風格讓你選 (a/b/c)：

(a) 條列式 — 「3 個重點，每點 1 句」
(b) 結構化 — 「先給結論再給依據，含 caveat」
(c) 草稿式 — 「直接給可用的 email/文案 draft，我自己再改」

或者告訴我「不要長這樣」的反例也行（例：不要太長 / 不要太正式）。
```

### 4. 給限制
> 「不要怎樣？例：不要太長 / 不要技術術語 / 不要 emoji / 不要太正式 / 不要直接給結論先給選項」

---

## 組合 + 執行

4 答案收齊後，組合成 prompt：

```
[Context]
我是 {部門} 主管，{1-2 句工作描述}。

[Your Role]
扮演 {角色}

[Task]
{用戶描述的 task}

[Example of good output]
{範例}

[Constraints]
- 不要 {限制 1}
- 不要 {限制 2}
```

跑這個 prompt 給 Claude（即自己，因為 skill 在 Claude session 內 — 直接接續對話 produce result）。

---

## 為什麼這樣設計

主管不會寫 prompt engineering，但會「派工」— 派工就是這四步：
- 餵 context = 帶下屬熟悉狀況
- 給角色 = 定 KPI
- 給範例 = 講樣子
- 給限制 = 講禁忌

所以這 skill 名 `delegate-sop` 不叫 `prompt-engineering`。

## 反模式

- ❌ 把四步當問卷一次列完 → 主管會放棄。一次問一個。
- ❌ 直接跑 prompt 沒組合 → 主管要看到自己派工的成果。
- ❌ 開頭就講「我們要做 prompt engineering」→ 違反 W1 教學原則。
